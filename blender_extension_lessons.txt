BLENDER EXTENSION DEVELOPMENT — LESSONS CARRIED FORWARD FROM THE MECHANISM LIBRARY PROJECT
=============================================================================================

Source: mechanisms_core (Blender addon: parametric gear/fastener/bearing/spring/
ratchet generators), docs/CONVENTIONS.md, and this project's Claude memory files.

This document extracts the parts of that project that are genuinely general —
true for any Blender extension built around bpy operators, custom properties,
and generated geometry — separated from the parts that were specific to
mechanism generation (gear math, thread nomenclature, etc.) and won't transfer.

Everything below was learned the hard way: found via a real bug, confirmed
empirically in Blender, then written down so it wouldn't need re-deriving.
Treat this as a starting "gotchas" doc for a new project, not a finished one —
it should keep growing the same way.


1. THE bpy.types.Operator PROPERTY SYSTEM HAS SHARP EDGES
-----------------------------------------------------------

- **An Operator cannot hold a PointerProperty to an ID type (Object, Mesh,
  etc.).** Blender silently drops that property AND EVERY PROPERTY DECLARED
  AFTER IT in the class on registration — no error, no warning, no console
  message. Properties just stop appearing in the panel. If you need a picker
  for an existing object (e.g. "pick a part to copy dimensions from"), put
  the PointerProperty on `bpy.types.WindowManager` (or `Scene`) instead, and
  have the operator read it from `context.window_manager.<name>`.

- **invoke() vs execute() have very different lifecycles.** `invoke()` runs
  ONCE, when the operator is freshly triggered (e.g. from a menu). `execute()`
  runs on that same initial call AND every time the user tweaks a property in
  the redo panel afterward. Any one-time setup (resetting a picker, seeding a
  random value, etc.) belongs in `invoke()`; putting it in `execute()` means
  it re-fires on every redo-panel edit and clobbers the user's own changes.

- **A property's `update` callback does not know which operator is
  "currently active."** If a property update needs to react differently
  depending on which operator invoked it (e.g. one shared WindowManager
  property used by several different operator types, each needing different
  behavior), the update callback can look up `context.active_operator` and
  branch on `op.bl_idname` — but see the headless-testing caveat below.

- **A PointerProperty's `poll` callback signature is `poll(self, object)` —
  no `context` parameter.** If the poll logic needs additional live context
  (e.g. "which operator is asking, and what's its current property state"),
  read `bpy.context.active_operator` directly inside the poll function body;
  it's not passed in, but it's available as a global.


2. A CROSS-OBJECT "MATCH TARGET" / "COPY DIMENSIONS FROM" PICKER — FULL RECIPE
--------------------------------------------------------------------------------

If a project needs "generate part B sized to fit part A" (a nut sized to a
bolt's thread, a bearing sized to a shaft's OD, a hinge pin sized to a
matching hole — anything where one generated object's stamped custom
properties should drive another's operator properties), this exact pattern
works and avoids several non-obvious pitfalls:

  a. Stamp custom properties on every generated object describing what
     another part would need to match it (e.g. `obj["my_diameter"] = 8.0`).
  b. Put a single `PointerProperty` picker on `WindowManager`, with a `poll`
     that accepts objects carrying the right stamped keys, and an `update`
     callback.
  c. Every participating operator implements its own `bmech_sync_target(self,
     context)` method (or similar) that reads the picker and copies whatever
     fields it drives from the target's stamped properties.
  d. The operator's `invoke()` resets the picker to `None` — never `execute()`
     (see the invoke/execute distinction above: resetting in `execute()`
     would wipe the user's pick on every redo-panel tweak).
  e. The `update` callback on the WindowManager property MUST treat a
     reset-to-`None` as a no-op and bail out immediately — do NOT let it
     trigger a sync/rebuild. Resetting the picker in `invoke()` (step d)
     itself fires this same `update` callback, and if it isn't special-cased,
     it will sync/rebuild using whatever `context.active_operator` still
     points at — which at that exact moment is the PREVIOUS object being
     edited, not the new one about to be created. Confirmed real bug: this
     deleted and rebuilt the previous part at the new 3D cursor position.
  f. To make a live pick actually rebuild the object in the viewport: delete
     whatever is currently selected+active (its own mesh data too, if
     orphaned), call `op.bmech_sync_target(context)`, then call
     `op.execute(context)` again directly. Do NOT reach for
     `bpy.ops.ed.undo()` to mimic what the redo panel does internally —
     confirmed to delete the object being edited without rebuilding it when
     called from inside a property update callback. This works because of a
     convention worth adopting generally: have every generator's `execute()`
     end by deselecting everything and selecting+activating exactly what it
     just built, so "whatever is selected+active" is always a reliable
     handle on "the thing this operator most recently produced."
  g. Freeze (grey out) only the specific fields the sync function actually
     drives, and only when the specific target carries the properties needed
     to drive them — not a blanket freeze of every matchable field whenever
     any target is picked. A target that doesn't stamp a given property
     should leave the corresponding field editable.
  h. Decide deliberately whether the target poll should be loose (accept any
     correctly-stamped object regardless of "kind") or restricted to a
     specific opposite/compatible kind. Both are legitimate depending on the
     domain: a loose poll is right when cross-kind matches can be physically
     meaningful (e.g. sharing a mounting hub between unrelated part types);
     a restricted poll is right when the domain has a hard physical
     constraint with no legitimate same-kind case (e.g. a male/female thread
     pairing — two male threads can never mate). Don't default to one
     without checking which situation you're in.
  i. THIS ENTIRE FEATURE CATEGORY IS INVISIBLE TO HEADLESS TESTING.
     `context.active_operator` is always `None` in `blender --background`
     scripts, and invoking an operator with `'INVOKE_DEFAULT'` collapses to
     calling `execute()` directly rather than dispatching to a custom
     `invoke()` — there's no real window/event system in background mode.
     None of the sync-on-pick, reset-on-invoke, or rebuild-on-pick behavior
     can be exercised by a headless script. Budget for manual GUI
     verification of these specific behaviors; a clean headless test run
     does not cover them. (What CAN be tested headlessly: the pure data
     functions — the sync function's field-copying logic, the poll's
     kind-checking logic, given a fake operator object with no real Blender
     operator context. Test those directly rather than trying to fake
     invoke/execute/context.active_operator.)


3. THE EXACT BOOLEAN SOLVER IS FRAGILE AT COINCIDENT SURFACES
-----------------------------------------------------------------

Blender's `EXACT` boolean solver (the one to use for precision mechanical
parts — `FAST` is not reliable enough for this kind of work) has specific,
learnable failure modes around surfaces that are supposed to touch exactly:

- **A cutter surface that's supposed to be flush with the target surface can
  be misread as a no-op** (a "hairline touch" rather than a genuine
  overlap), leaving material uncut, OR can produce genuine non-manifold
  defects. Two different, equally valid fixes exist for two different kinds
  of coincidence — know which one applies before reaching for either:
    - If the coincidence is a REAL dimensional target (e.g. a bore that
      must be exactly some diameter): cut in the right order (the cut that
      "opens up" the final size goes last) and pad that cutter by a tiny,
      deliberate amount past the exact dimension (a hairline OVER, not
      under) rather than trying to hit the dimension exactly. Getting the
      padding DIRECTION wrong (undersizing when you should oversize, or
      vice versa) can produce a real, physical, non-cosmetic defect (e.g.
      an uncut lip) rather than just a mesh-topology glitch — always confirm
      which direction actually closes the gap by testing both.
    - If the coincidence is a PHASE/WIND-UP issue (e.g. a helical cutter
      whose repeating profile happens to start at the same phase as the
      real boundary it's cutting against): the fix is axial/positional
      overshoot so the cutter's pattern is already "wound up" into a
      steady-state repeat before it reaches the real boundary — not
      resizing. The overshoot AMOUNT can matter non-trivially: the most
      "obviously safe"-looking overshoot (a full repeat period) can
      accidentally reproduce the exact same problematic phase at the real
      boundary that a smaller, specifically-chosen overshoot (e.g. half a
      period) avoids. Test the actual overshoot amount, don't assume more
      is automatically safer.
    - A final `bmesh.ops.remove_doubles` ("Merge by Distance") pass with a
      small, fixed tolerance is a good general cleanup for whatever genuine
      coincident geometry remains after cuts are ordered/padded correctly —
      but it can't fix a torn hole (missing face / real non-manifold
      boundary edge) — only actual duplicate geometry.
- **A boolean UNION of two independently-built curved surfaces is riskier
  than a boolean DIFFERENCE of a cutter against solid material.** A plain
  difference against a solid blank is numerically easy for the solver — the
  cutter naturally overlaps real material throughout. A union of two
  separately-built, nearly-coincident surfaces is a much harder case. If a
  design can be restructured as "build one solid, subtract everything,
  union on the last simple/blocky piece" instead of unioning multiple
  organic shapes together, prefer that ordering — and do the riskiest step
  first so failures surface before the cheap reliable steps run.
- **Re-run `recalc_face_normals` after a UNION even on a topologically valid
  result.** EXACT-solver unions of complex meshes can come back with
  inconsistent face winding even when the mesh is genuinely manifold and
  solid — it will render as if it has holes. Differences haven't shown this
  in practice, but if a difference ever shows the same symptom, this is the
  first thing to try.


4. bmesh CONSTRUCTION PATTERNS THAT AVOID DEGENERATE GEOMETRY
------------------------------------------------------------------

- **Chain multi-segment solids of revolution through SHARED vertices at
  each junction, rather than stacking independently-capped primitives end
  to end.** Two independently-capped segments that happen to meet at the
  same radius leave DUPLICATE coincident faces there (each segment's own
  end cap covering the identical disk) — a fragile degenerate case for the
  boolean solver. A single continuous solid built by walking through a list
  of (radius, z) checkpoints and connecting each consecutive pair with
  shared vertices has no internal face at any junction, so the problem
  can't arise structurally.
- **Cap a closed profile loop with a single n-gon face, not a fan from an
  added center vertex, whenever the profile can contain near-collinear
  points through the center.** A profile with any two points that end up
  radially aligned with the origin (common with any profile that has a
  straight radial segment somewhere, not just this project's specific gear
  math) degenerates a fan-to-center triangle to exactly zero area at that
  point. A single n-gon face has no per-point triangles and can't hit this
  case regardless of profile shape. If the region ISN'T a simple closed
  loop but an ANNULUS (bridging an inner and outer loop, e.g. a ring shape),
  use `bmesh.ops.triangle_fill` fed the boundary edges of BOTH loops
  together in one call — it triangulates the annular region directly,
  treating the inner loop as a hole — rather than hand-bridging the two
  loops with an index-matched strip of quads (which reintroduces the same
  collinear-point degenerate case if the two loops don't have a 1:1
  angle-matched point correspondence, which they usually won't for
  independently-built loops).
- **A zero-area face is not always harmless, and dropping it is not always
  safe either.** A degenerate zero-area triangle can still be edge-manifold
  (it shares its short edge with exactly one real neighboring face) —
  cosmetically ugly but not a hard defect. Simply deleting it, instead of
  restructuring the cap to avoid creating it in the first place, can turn
  that edge into a genuine non-manifold boundary. Fix the construction that
  creates the degenerate face, don't patch around it after the fact.


5. GEOMETRY THAT DEPENDS ON A ROLLING/ORBITING RELATIONSHIP NEEDS THE RIGHT
   KINEMATIC MODEL, NOT JUST THE OBVIOUS RATIO
-------------------------------------------------------------------------------

Any time a generated part's rotation/position must be derived from another
part's geometry via a rolling-contact relationship (gear-on-gear, gear-on-
rack, a planet gear orbiting a sun gear, a wheel rolling along a track,
etc.), the "obvious" naive ratio is a common place to get a sign or a
missing term wrong, and the resulting bug can look like it "mostly works":

- Two objects/features in ROLLING CONTACT at a single point must have equal
  and opposite (or equal and same-direction, depending on internal vs.
  external contact) surface velocity/displacement AT THAT POINT — this is
  the one relationship to derive from directly, rather than guessing at a
  ratio and a sign by analogy with a similar-looking case. Two cases that
  look structurally similar (e.g. two side-by-side external gears, vs. one
  external part contacting a second part at a 90°-different relative
  angular position) can have genuinely DIFFERENT required signs, because
  the relative CONTACT ANGLE — not just "is it internal or external contact"
  — determines the sign. Don't assume a rule proven for one relative
  geometry (e.g. two objects side-by-side) transfers unchanged to a
  differently-positioned case (e.g. one object directly below/inside
  another) just because both are "the same kind of contact" in some coarser
  sense.
- An object whose own reference frame is simultaneously ORBITING (its center
  moves around some other point) AND SPINning about its own axis picks up
  an extra term beyond the naive tooth/feature-count ratio — the classic
  "coin rotation paradox" (a coin of radius r rolling once, without
  slipping, around the OUTSIDE of a same-size fixed coin spins TWICE, not
  once, because its own axis completes one extra full revolution around the
  fixed coin in addition to the rolling-contact spin). Any planetary/
  epicyclic-style mechanism (not just gears) has this term; leaving it out
  produces a formula that happens to look plausible and even builds without
  error, while being wrong for most parameter combinations.
- If a design has an "assembly/spacing condition" (e.g. "N objects can only
  be placed evenly around a circle if some combination of tooth/feature
  counts satisfies a divisibility rule") — that condition usually exists
  specifically because it's the number-theoretic requirement for a SINGLE
  placement formula (correct for one instance) to ALSO be correct
  simultaneously for every other evenly-spaced instance. If a fix to the
  placement formula still fails for multiple instances even though the
  spacing condition is satisfied, suspect the placement formula itself
  first, not the spacing condition.


6. VERIFICATION METHODOLOGY — THE SINGLE MOST VALUABLE HABIT FROM THIS PROJECT
-----------------------------------------------------------------------------------

- **"It builds without a Python exception" is not "it's correct."** Every
  geometry generator in this codebase that was verified only by "does the
  operator return FINISHED without crashing" turned out, at some point, to
  hide a real defect (non-manifold edges, zero-area faces, wrong-signed
  rotation, gross physical interference between parts) that a slightly
  deeper headless check caught immediately. Build a small headless test
  harness EARLY and keep reaching for it:
    - Non-manifold edge count: `sum(1 for e in bm.edges if not e.is_manifold)`.
    - Zero-area face count: `sum(1 for f in bm.faces if f.calc_area() < 1e-9)`.
    - For "does part A actually clash with part B" questions (does this
      rotation/position combination produce real interference, not just a
      valid mesh): duplicate one part, add a `BOOLEAN` modifier set to
      `INTERSECT` against the other part, apply it, and read the resulting
      volume via `bm.calc_volume(signed=False)`. Zero (or floating-point
      noise, ~1e-14) means no interference; anything meaningfully above
      zero (calibrate the threshold against the actual feature size — e.g.
      compare to a single tooth's approximate volume) means real
      interference. This is a far more reliable way to validate a
      rotation/positioning formula than re-checking the symbolic derivation
      a second time — trust the empirical result over a re-derivation that
      "looks right."
    - When checking an object with a LIVE (unapplied) modifier, evaluate it
      via `obj.evaluated_get(depsgraph).to_mesh()` — checking `obj.data`
      directly only shows the pre-modifier mesh and can produce a
      completely misleading "defect" (e.g. a flat, single-sided profile
      that's expected to look "non-manifold" before its Solidify modifier
      runs) or hide a real one.
  - **Before concluding a fix caused a regression, confirm the "regression"
    doesn't already exist in the pre-fix code.** `git stash`, re-run the
    exact same failing check against the unmodified file, `git stash pop`.
    Several apparent regressions in this project turned out to be
    pre-existing, parameter-combination-specific defects completely
    unrelated to whatever was just changed — confirmed identical failure
    counts/messages before and after. This takes under a minute and
    prevents chasing a phantom cause.
  - **When re-deriving a formula from scratch under time pressure (or
    at all), sanity-check it against ONE fully-worked-out concrete numeric
    example by hand before trusting it broadly**, especially when an
    existing, independent piece of logic (like a pre-existing centering/
    positioning formula) interacts with the new one. A first-pass version
    of a "nice to have" cosmetic phase-alignment feature in this project
    computed a plausible-looking shift that was actually WRONG for roughly
    half of all parameter combinations, because it didn't account for an
    existing formula's own parity-dependent behavior — caught only by
    working through one concrete example by hand before shipping, not by
    the formula "looking reasonable."
  - **If the extension is installed as a separate copy from the repo being
    edited (rather than a live symlink), ALWAYS confirm the installed copy
    actually matches the repo file before trusting a test result** — hash-
    compare them. Reload Scripts (F3) and even relaunching Blender only
    reload whatever is in the installed copy; neither one detects or pulls
    in repo edits automatically if the install is a real file copy instead
    of a symlink/junction. A stale installed copy produces a very
    convincing-looking "the fix doesn't work in Blender even though the
    headless test is clean" false alarm. Check this FIRST, before assuming
    the fix itself is wrong, any time headless and live-GUI results seem to
    disagree.


7. DOCUMENTATION ARCHITECTURE THAT ACTUALLY GETS MAINTAINED
------------------------------------------------------------------

- **One project-wide "conventions" doc as the single source of truth**,
  covering rules that apply everywhere (units, compensation-direction
  conventions, validation severity levels, any cross-cutting pattern like
  the Match Target system above) — plus **one README per logical
  sub-family/module group** covering what's shared within that group, plus
  **one doc per individual generator/feature** covering only what's unique
  to it. Anything that would otherwise be repeated identically across many
  individual docs belongs in the shared level instead — that's specifically
  what keeps N copies of the same convention from silently drifting apart
  as the codebase evolves.
- **Write the doc in the same change that adds or edits the code**, not as
  separate follow-up work — a stale doc describing behavior that no longer
  exists is worse than no doc, since it actively misleads rather than just
  being unhelpful.
- **In-code comments should explain WHY, not WHAT** — a hidden constraint,
  a subtle invariant, a workaround for a specific solver/API quirk,
  behavior that would surprise a future reader. Well-named
  functions/variables already say what the code does; comments repeating
  that add noise without adding information. The most valuable comments in
  this codebase are the ones documenting a REJECTED alternative approach
  and why it failed empirically — that's exactly the information a future
  editor (human or AI) needs to avoid re-attempting something already ruled
  out, and it's the information most likely to be lost if not written down.


8. KNOWN GAP — NOT COVERED HERE, FLAGGED FOR THE NEW PROJECT TO BUILD FRESH
--------------------------------------------------------------------------------

This project never touched Blender's node systems (geometry nodes, shader
nodes, compositor nodes) — everything here is bmesh/bpy.ops mesh generation
and boolean modeling. Scripting node trees via bpy is reportedly a weak spot
for LLM-generated code in general, plausibly because Blender overhauled the
node-group interface API in version 4.0 (the old `NodeSocketInterface`/
`node_tree.inputs` system was replaced by a unified `node_tree.interface`
with `new_socket()`), and a lot of existing tutorials/answers/training data
still reflect pre-4.0 syntax — producing confidently-wrong hybrid code that
mixes old and new API calls, rather than a clean failure. If continuing this
work into node-scripting, expect to build an equivalent gotchas section from
scratch (version-specific socket/interface API differences, node type
idnames, manual node `.location` layout since there's no auto-arrange,
`links.new()` patterns, group input/output node quirks) — don't assume
anything above transfers to that domain.


9. WHAT DID NOT MAKE IT INTO THIS DOCUMENT (DELIBERATELY)
------------------------------------------------------------

Mechanism-specific domain knowledge — involute gear tooth math, ISO thread
nomenclature, planetary gear tooth-count assembly conditions, and similar —
was excluded on purpose. It was real, hard-won, and correctly documented in
this project's own docs/ and memory files, but it doesn't generalize to
"Blender extensions" as a category the way everything above does. A future
project in an unrelated domain should expect to rebuild that layer of
domain-specific knowledge from scratch, the same way this project did.
